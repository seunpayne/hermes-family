# Generating Prisma Migrations Without a Running Database

When building a project scaffold where no PostgreSQL instance is available
locally (e.g. the Docker container doesn't have one), use Prisma's
standalone migration diff tool to generate the SQL files.

## Technique: `prisma migrate diff`

```bash
mkdir -p prisma/migrations/0001_init
npx prisma migrate diff \
  --from-empty \
  --to-schema-datamodel prisma/schema.prisma \
  --script > prisma/migrations/0001_init/migration.sql
```

This produces a complete DDL file with all:
- Custom enum types
- CREATE TABLE statements with columns, defaults, nullability
- Foreign key constraints with ON DELETE CASCADE/RESTRICT
- Composite unique constraints and multi-column indexes
- Standalone single-column indexes

## Migration Metadata

Prisma also needs a migration lock file:

```toml
# prisma/migrations/migration_lock.toml
# Please do not edit this file manually
# It should be added in your version-control system (i.e. Git)
provider = "postgresql"
```

## Applying Later

When a PostgreSQL target becomes available:

```bash
# Set DATABASE_URL in environment
export DATABASE_URL="postgresql://user:pass@host:5432/streetwise"

# Apply pending migrations
npx prisma migrate deploy
```

## Adding to an Existing Schema (No DB Access)

When you add a new model to an existing schema but can't connect to the database (e.g. Docker/dev environment without local Postgres, or pooler-only Supabase):

### 1. Create the migration directory and SQL file

```bash
TIMESTAMP=$(date -u +%Y%m%d%H%M%S)
mkdir -p prisma/migrations/${TIMESTAMP}_add_new_model
```

Write `migration.sql` in that directory following Prisma's naming conventions. Use `gen_random_uuid()::text` for default IDs (PostgreSQL), `TEXT` for String, `TIMESTAMPTZ` for DateTime:

```sql
-- Migration: add_new_model
CREATE TABLE IF NOT EXISTS "NewModel" (
    id TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
    "fieldName" TEXT NOT NULL,
    "otherField" INTEGER NOT NULL DEFAULT 0,
    "createdAt" TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE ("fieldName"),
    UNIQUE ("residentId", "auditLogId")  -- composite unique
);

CREATE INDEX IF NOT EXISTS "NewModel_fieldName_idx" ON "NewModel" ("fieldName");
```

### 2. Regenerate Prisma client types

```bash
npx prisma generate
```

This updates `node_modules/@prisma/client` with the new model so TypeScript compilation succeeds.

### 3. Verify compilation

```bash
npx tsc --noEmit  # Should show no new errors for the new model
npm run build      # Full build
```

### 4. Apply on deploy

When deploying to staging/production where a real database exists:

```bash
npx prisma migrate deploy
```

### Why This Works

- Prisma's migration tracking is file-based — it reads `prisma/migrations/` directory and applies any migration that hasn't been applied yet
- `prisma generate` creates TypeScript types from the schema only — it doesn't need a database
- The migration SQL is committed to version control and applied during CI/deployment
- This avoids the chicken-and-egg problem of needing a database to generate migration files when the only available database is production

## Why This Matters

- No database server needed at scaffold time
- Migration SQL is committed to version control
- Reproducible — same SQL applies to staging and production
- No schema drift between local dev, staging, and production
- Avoids `prisma db push` which doesn't create migration files

## Prisma Schema Validation (without DB)

Prisma's `validate` command requires DATABASE_URL even for schema-only
checks. Instead, verify schema correctness by running:

```bash
npx prisma generate   # Will fail on schema errors
npm run build          # Will fail if Prisma types are wrong
```

If generate succeeds and build succeeds, the schema is valid.
